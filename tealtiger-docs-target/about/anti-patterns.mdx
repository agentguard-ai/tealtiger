## Purpose

This document describes **anti‑patterns** — common but incorrect ways teams attempt to use TealTiger.

Avoiding these patterns is critical to:
- Maintain deterministic behavior
- Prevent performance degradation
- Preserve policy clarity
- Ensure long‑term scalability and trust

These are **not bugs** in TealTiger — they are **misaligned usage patterns**.

---

## Anti‑Patterns to Avoid

### 1. Using TealTiger as a Business Logic Engine

**Anti‑pattern:**  
Embedding core application logic inside TealTiger policies.

**Why this is wrong:**
- Policies are for **governance**, not domain logic
- Business logic changes frequently; policies should not
- Increases coupling and reduces clarity

**Correct approach:**  
Keep business logic in your application.  
Use TealTiger only to **enforce constraints** and **guardrails** around AI behavior.

---

### 2. Treating Policies as Dynamic, Frequently Mutating Rules

**Anti‑pattern:**  
Updating policies on every deploy or runtime event.

**Why this is wrong:**
- Breaks policy stability guarantees
- Makes audit trails harder to reason about
- Increases risk of unintended behavior changes

**Correct approach:**  
Policies should be:
- Versioned
- Reviewed
- Changed deliberately

Runtime context should flow through **inputs**, not policy rewrites.

---

### 3. Using TealTiger as a Real‑Time Monitoring or Alerting System

**Anti‑pattern:**  
Expecting TealTiger to:
- Generate alerts
- Trigger paging
- Act as an observability dashboard

**Why this is wrong:**
- TealTiger emits **events**, it does not manage alerts
- Alert fatigue and correlation belong in observability tools

**Correct approach:**  
Forward audit events to:
- SIEM
- Logging platforms
- Observability pipelines

---

### 4. Relying on TealTiger for Secret or Credential Handling

**Anti‑pattern:**  
Passing secrets, API keys, or credentials into policies or metadata.

**Why this is wrong:**
- Violates separation of concerns
- Increases blast radius
- Creates audit and compliance risks

**Correct approach:**  
Use:
- IAM
- KMS
- Secret managers

Pass only **identity references or roles** into TealTiger.

---

### 5. Using Policies to “Fix” Poor Agent or Prompt Design

**Anti‑pattern:**  
Attempting to compensate for:
- Unbounded prompts
- Poor agent instructions
- Unsafe tool access

by adding increasingly complex policies.

**Why this is wrong:**
- Governance cannot replace good design
- Leads to brittle enforcement
- Masks root causes

**Correct approach:**  
Design agents and prompts responsibly first.  
Use TealTiger to **validate**, not **repair**, AI behavior.

---

### 6. Encoding Vendor‑Specific Logic Directly into Policies

**Anti‑pattern:**  
Hard‑coding OpenAI, Anthropic, Bedrock, or other vendor behaviors into policies.

**Why this is wrong:**
- Reduces portability
- Locks policies to infrastructure decisions
- Makes migrations painful

**Correct approach:**  
Policies should be:
- Vendor‑agnostic
- Based on behavior, intent, and risk
- Decoupled from provider implementation details

---

### 7. Expecting Probabilistic or “AI‑Like” Policy Decisions

**Anti‑pattern:**  
Assuming policies will:
- Learn over time
- Make fuzzy or probabilistic decisions
- Self‑adapt without changes

**Why this is wrong:**
- TealTiger is deterministic by design
- Predictability is a security requirement

**Correct approach:**  
If a decision must change, **update the policy**.  
Determinism is a feature, not a limitation.

---

### 8. Treating Audit Logs as Debug Logs

**Anti‑pattern:**  
Using audit events as verbose debugging output.

**Why this is wrong:**
- Audit logs must remain structured and stable
- Excessive noise reduces forensic value
- Increases storage and processing cost

**Correct approach:**  
Use:
- Application logs for debugging
- TealTiger audit logs for governance and traceability

---

### 9. Centralizing All Governance Decisions in a Single Policy

**Anti‑pattern:**  
Creating one massive, monolithic policy covering:
- Cost
- Security
- Reliability
- Compliance
- Environment logic

**Why this is wrong:**
- Hard to review
- Hard to test
- Hard to evolve safely

**Correct approach:**  
Compose policies by concern:
- Cost policies
- Security policies
- Reliability policies

Evaluate them independently but coherently.

---

### 10. Using TealTiger as a “Set and Forget” Control

**Anti‑pattern:**  
Deploying TealTiger once and never reviewing policies again.

**Why this is wrong:**
- AI systems evolve
- Usage patterns change
- Risk profiles shift

**Correct approach:**  
Treat policies as **living governance artifacts**:
- Review periodically
- Adjust with intent
- Maintain version history

---

## Design Principle Reinforced

TealTiger works best when it is:

- Explicit
- Deterministic
- Composable
- Boring (in the best way)

If policies feel clever, magical, or opaque — that is a warning sign.

---

## Relationship to Non‑Goals

- **Non‑Goals** define what TealTiger does *not* attempt to solve
- **Anti‑Patterns** define how TealTiger should *not* be used

Together, they form the **guardrails for correct adoption**.

---

## Summary

Avoiding these anti‑patterns ensures:
- Predictable enforcement
- Clean audit trails
- Scalable governance
- Developer trust

TealTiger is a **governance primitive**, not a replacement for good engineering discipline.
